172  * \defgroup ParallelGroup Parallelization
173  * \brief Functions, classes and documentation related to parallelization and
174  * Charm++
175 
176 SpECTRE builds a layer on top of Charm++ that performs various safety checks and
177 initialization for the user that can otherwise lead to difficult-to-debug
178 undefined behavior. The central concept is what is called a %Parallel
179 Component. A %Parallel Component is a struct with several type aliases that
180 is used by SpECTRE to set up the Charm++ chares and allowed communication
181 patterns. %Parallel Components are input arguments to the compiler, which then
182 writes the parallelization infrastructure that you requested for the executable.
183 There is no restriction on the number of %Parallel Components, though
184 practically it is best to have around 10 at most.
185 
186 Here is an overview of what is described in detail in the sections below:
187 
188 - Metavariables: Provides high-level configuration to the compiler, e.g. the
189  physical system to be simulated.
190 - Phase: Defines distinct simulation phases separated by a global
191  synchronization point, e.g. `Initialize`, `Evolve` and `Exit`.
192 - Algorithm: In each phase, iterates over a list of actions until the current
193  phase ends.
194 - %Parallel component: Maintains and executes its algorithm.
195 - Action: Performs a computational task, e.g. evaluating the right hand side of
196  the time evolution equations. May require data to be received from another
197  action potentially being executed on a different core or node.
198 
199 ### The Metavariables Class
200 
201 SpECTRE takes a different approach to input options passed to an executable than
202 is common. SpECTRE not only reads an input file at runtime but also has many
203 choices made at compile time. The compile time options are specified by what is
204 referred to as the metavariables. What exactly the metavariables struct
205 specifies depends somewhat on the executable, but all metavariables structs must
206 specify the following:
207 
208 - `help`: a `static constexpr OptionString` that will be printed as part of the
209  help message. It should describe the executable and basic usage of it, as well
210  as any non-standard options that must be specified in the metavariables and
211  their current values. An example of a help string for one of the testing
212  executables is:
213  \snippet Test_AlgorithmCore.cpp help_string_example
214 - `component_list`: a `tmpl::list` of the parallel components (described below)
215  that are to be created. Most evolution executables will have the
216  `DgElementArray` parallel component listed. An example of a `component_list`
217  for one of the test executables is:
218  \snippet Test_AlgorithmCore.cpp component_list_example
219 - `Phase`: an `enum class` that must contain at least `Initialization` and
220  `Exit`. Phases are described in the next section.
221 - `determine_next_phase`: a static function with the signature
222  \code
223  static Phase determine_next_phase(
224  const Phase& current_phase,
225  const Parallel::CProxy_ConstGlobalCache<EvolutionMetavars>& cache_proxy)
226  noexcept;
227  \endcode
228  What this function does is described below in the discussion of phases.
229 
230 There are also several optional members:
231 
232 - `input_file`: a `static constexpr OptionString` that is the default name of
233  the input file that is to be read. This can be overridden at runtime by
234  passing the `--input-file` argument to the executable.
235 - `ignore_unrecognized_command_line_options`: a `static constexpr bool` that
236  defaults to `false`. If set to `true` then unrecognized command line options
237  are ignored. Ignoring unrecognized options is generally only necessary for
238  tests where arguments for the testing framework, Catch, are passed to the
239  executable.
240 
241 ### Phases of an Execution
242 
243 Global synchronization points, where all cores wait for each other, are
244 undesirable for scalability reasons. However, they are sometimes inevitable for
245 algorithmic reasons. That is, in order to actually get a correct solution you
246 need to have a global synchronization. SpECTRE executables can have multiple
247 phases, where after each phase a global synchronization occurs. By global
248 synchronization we mean that no parallel components are executing or have more
249 tasks to execute: everything is waiting on a task to perform.
250 
251 Every executable must have at least two phases, `Initialization` and `Exit`. The
252 next phase is decided by the static member function `determine_next_phase` in
253 the metavariables. Currently this function has access to the phase that is
254 ending, and also the global cache. In the future we will add support for
255 receiving data from various components to allow for more complex decision
256 making. Here is an example of a `determine_next_phase` function and the `Phase`
257 enum class:
258 \snippet Test_AlgorithmCore.cpp determine_next_phase_example
259 
260 In contrast, an evolution executable might have phases `Initialization`,
261 `SetInitialData`, `Evolve`, and `Exit`, but have a similar `switch` or `if-else`
262 logic in the `determine_next_phase` function. The first phase that is entered is
263 always `Initialization`. During the `Initialization` phase the `initialize`
264 function is called on all parallel components. Once all parallel components'
265 `initialize` function is complete, the next phase is determined and the
266 `execute_next_phase` function is called after on all the parallel components.
267 
268 At the end of an execution the `Exit` phase has the executable wait to make sure
269 no parallel components are performing or need to perform any more tasks, and
270 then exits. An example where this approach is important is if we are done
271 evolving a system but still need to write data to disk. We do not want to exit
272 the simulation until all data has been written to disk, even though we've
273 reached the final time of the evolution.

275 ### The Algorithm
276 
277 Since most numerical algorithms repeat steps until some criterion such as the
278 final time or convergence is met, SpECTRE's parallel components are designed to
279 do such iterations for the user. An Algorithm executes an ordered list of
280 actions until one of the actions cannot be evaluated, typically because it is
281 waiting on data from elsewhere. When an algorithm can no longer evaluate actions
282 it passively waits by handing control back to Charm++. Once an algorithm
283 receives data, typically done by having another parallel component call the
284 `receive_data` function, the algorithm will try again to execute the next
285 action. If the algorithm is still waiting on more data then the algorithm will
286 again return control to Charm++ and passively wait for more data. This is
287 repeated until all required data is available. The actions that are iterated
288 over by the algorithm are called iterable actions and are described below.
289 
290 \note
291 Currently all Algorithms must execute the same actions (described below) in all
292 phases. This restriction is also planned on being relaxed if the need arises.
293 
294 ### %Parallel Components
295 
296 Each %Parallel Component struct must have the following type aliases:
297 1. `using chare_type` is set to one of:
298  1. `Parallel::Algorithms::Singleton`s have one object in the entire execution
299  of the program.
300  2. `Parallel::Algorithms::Array`s hold zero or more elements, each of which
301  is an object distributed to some core. An array can grow and shrink in
302  size dynamically if need be and can also be bound to another array. A
303  bound array has the same number of elements as the array it is bound to,
304  and elements with the same ID are on the same core. See Charm++'s chare
305  arrays for details.
306  3. `Parallel::Algorithms::Group`s are arrays with
307  one element per core which are not able to be moved around between
308  cores. These are typically useful for gathering data from array elements
309  on their core, and then processing or reducing the data further. See
310  [Charm++'s](http://charm.cs.illinois.edu/help) group chares for details.
311  4. `Parallel::Algorithms::Nodegroup`s are similar to
312  groups except that there is one element per node. For Charm++ SMP (shared
313  memory parallelism) builds, a node corresponds to the usual definition of
314  a node on a supercomputer. However, for non-SMP builds nodes and cores are
315  equivalent. We ensure that all entry method calls done through the
316  Algorithm's `simple_action` and `receive_data` functions are
317  threadsafe. User controlled threading is possible by calling the non-entry
318  method member function `threaded_simple_action`.
319 2. `using metavariables` is set to the Metavariables struct that stores the
320  global metavariables. It is often easiest to have the %Parallel
321  Component struct have a template parameter `Metavariables` that is the
322  global metavariables struct. Examples of this technique are given below.
323 3. `using action_list` is set to a `tmpl::list` of the %Actions (described
324  below) that the Algorithm running on the %Parallel Component executes. The
325  %Actions are executed in the order that they are given in the `tmpl::list`.
326 4. `using initial_databox` is set to the type of the DataBox that will be passed
327  to the first Action of the `action_list`. Typically it is the output of some
328  simple action called during the `Initialization` Phase.
329 5. `using options` is set to a (possibly empty) `tmpl::list` of the option
330  structs. The options are read in from the input file specified in the main
331  `Metavariables` struct. After being read in they are passed to the
332  `initialize` function of the parallel component, which is described below.
333 6. `using const_global_cache_tag_list` is set to a `tmpl::list` of CacheTags
334  that are required by the parallel component. Some examples are
335  `CacheTags::TimeStepper` for evolutions, or `CacheTags::FinalTime`.
336 
337 \note Array parallel components must also specify the type alias `using
338 array_index`, which is set to the type that indexes the %Parallel Component
339 Array. Charm++ allows arrays to be 1 through 6 dimensional or be indexed by a
340 custom type. The Charm++ provided indexes are wrapped as
341 `Parallel::ArrayIndex1D` through `Parallel::ArrayIndex6D`. When writing custom
342 array indices, the [Charm++ manual](http://charm.cs.illinois.edu/help) tells you
343 to write your own `CkArrayIndex`, but we have written a general implementation
344 that provides this functionality; all that you need to provide is a
345 plain-old-data
346 ([POD](http://en.cppreference.com/w/cpp/concept/PODType)) struct of the size of
347 at most 3 integers.
348 
349 %Parallel Components have a static `initialize` function that is used
350 effectively as the constructor of the components. The signature of the
351 initialize functions must be:
352 \code
353 static void initialize(
354  Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache, opts...);
355 \endcode
356 The `initialize` function is called by the Main %Parallel Component when
357 the execution starts and will typically call a simple %Action
358 to set up the initial state of the Algorithm, similar to what a constructor
359 does for classes. The `initialize` function also receives arguments that
360 are read from the input file which were specified in the `options` typelist
361 described above. The options are usually used to initialize the %Parallel
362 Component's DataBox, or even the component itself. An example of initializing
363 the component itself would be using the value of an option to control the size
364 of the %Parallel Component Array. The `initialize` functions of different
365 %Parallel Components are called in random order and so it is not safe to have
366 them depend on each other.
367 
368 Each parallel component must also decide what to do in the different phases of
369 the execution. This is controlled by an `execute_next_phase` function with
370 signature:
371 \code
372 static void execute_next_phase(
373  const typename metavariables::Phase next_phase,
374  const Parallel::CProxy_ConstGlobalCache<metavariables>& global_cache);
375 \endcode
376 The `determine_next_phase` function in the Metavariables determines the next
377 phase, after which the `execute_next_phase` function gets called. The
378 `execute_next_phase` function determines what the %Parallel Component should do
379 during the next phase. For example, it may simply call `perform_algorithm`, call
380 a series of simple actions, perform a reduction over an Array, or not do
381 anything at all. Note that `perform_algorithm` performs the same actions (the
382 ones in `action_list`) no matter what Phase it is called in.
383 
384 An example of a singleton %Parallel Component is:
385 \snippet Test_AlgorithmParallel.cpp singleton_parallel_component
386 
387 An example of an array %Parallel Component is:
388 \snippet Test_AlgorithmParallel.cpp array_parallel_component
389 Elements are inserted into the Array by using the Charm++ `insert` member
390 function of the CProxy for the array. The `insert` function is documented in
391 the Charm++ manual. In the above Array example `array_proxy` is a `CProxy` and
392 so all the documentation for Charm++ array proxies applies. SpECTRE always
393 creates empty Arrays with the constructor and requires users to insert however
394 many elements they want and on which cores they want them to be placed. Note
395 that load balancing calls may result in Array elements being moved.

397 ### %Actions
398 
399 For those familiar with Charm++, actions should be thought of as effectively
400 being entry methods. They are functions that can be invoked on a remote object
401 (chare/parallel component) using a `CProxy` (see the [Charm++
402 manual](http://charm.cs.illinois.edu/help)), which is retrieved from the
403 ConstGlobalCache using the parallel component struct and the
404 `Parallel::get_parallel_component()` function. %Actions are structs with a
405 static `apply` method and come in three variants: simple actions, iterable
406 actions, and reduction actions. One important thing to note
407 is that actions cannot return any data to the caller of the remote method.
408 Instead, "returning" data must be done via callbacks or a callback-like
409 mechanism.
410 
411 The simplest signature of an `apply` method is for iterable actions:
412 \snippet Test_AlgorithmCore.cpp apply_iterative
413 The return type is discussed at the end of each section describing a particular
414 type of action. Simple actions can have additional arguments but must have at
415 least the arguments shown above. Reduction actions must have the above arguments
416 and an argument taken by value that is of the type the reduction was made over.
417 The `db::DataBox` should be thought of as the member data of the parallel
418 component while the actions are the member functions. The combination of a
419 `db::DataBox` and actions allows building up classes with arbitrary member data
420 and methods using template parameters and invocation of actions. This approach
421 allows us to eliminate the need for users to work with Charm++'s interface
422 files, which can be error prone and difficult to use.
423 
424 The ConstGlobalCache is passed to each action so that the action has access
425 to global data and is able to invoke actions on other parallel components. The
426 `ParallelComponent` template parameter is the tag of the parallel component that
427 invoked the action. A proxy to the calling parallel component can then be
428 retrieved from the ConstGlobalCache. The remote entry method invocations are
429 slightly different for different types of actions, so they will be discussed
430 below. However, one thing that is disallowed for all actions is calling an
431 action locally from within an action on the same parallel component.
432 Specifically,
433

436 Here `ckLocal()` is a Charm++ provided method that returns a pointer to the
437 local (currently executing) parallel component. See the [Charm++
438 manual](http://charm.cs.illinois.edu/help) for more information.
439 However, you are able to queue a new action to be executed later on the same
440 parallel component by getting your own parallel component from the
441 ConstGlobalCache (`Parallel::get_parallel_component<ParallelComponent>(cache)`).
442 The difference between the two calls is that by calling an action through the

444 when they are complete Charm++ will call the next queued action.

468 #### 1. Simple %Actions
469 
470 Simple actions are designed to be called in a similar fashion to member
471 functions of classes. They are the direct analog of entry methods in Charm++
472 except that the member data is stored in the `db::DataBox` that is passed in as
473 the first argument. There are a couple of important things to note with simple
474 actions:
475 
476 1. A simple action must return void but can use `db::mutate` to change values
477  of items in the DataBox if the DataBox is taken as a non-const reference.
478  There is one exception: if the input DataBox is empty, then the
479  simple action can return a DataBox of type `initial_databox`. That is, an
480  action taking an empty DataBox and returning the `initial_databox` is
481  effectively constructing the DataBox in its initial state.
482 2. A simple action is instantiated once for an empty
483  `db::DataBox<tmpl::list<>>`, once for a DataBox of type
484  `initial_databox` (listed in the parallel component), and once for each
485  returned DataBox from the iterable actions in the `action_list` in the
486  parallel component. In some cases you will need specific items to be in the
487  DataBox otherwise the action won't compile. To restrict which DataBoxes can
488  be passed you should use `Requires` in the action's `apply` function
489  template parameter list. For example,
490  \snippet Test_AlgorithmCore.cpp requires_action
491  where the conditional checks if any element in the parameter pack `DbTags` is
492  `CountActionsCalled`.
493 
494

496 from the ConstGlobalCache as follows:
497

500 If the simple action takes arguments then the arguments must be passed to the
501 `simple_action` method as a `std::tuple` (because Charm++ doesn't yet support
502 variadic entry method templates). For example,

511 the action is expecting to receive a `size_t` as its extra argument. Instead,
512 you can get a `std::tuple<size_t>` in one of two ways. First, you can pass in
513 `std::tuple<size_t>(0)`, second you can include the header
514 `Utilities/Literals.hpp` and then pass in `std::make_tuple(0_st)`.

516 #### 2. Iterable %Actions
517 
518 %Actions in the algorithm that are part of the `action_list` are
519 executed one after the other until one of them cannot be evaluated. Iterable
520 actions may have an `is_ready` method that returns `true` or `false` depending
521 on whether or not the action is ready to be evaluated. If no `is_ready` method
522 is provided then the action is assumed to be ready to be evaluated. The
523 `is_ready` method typically checks that required data from other parallel
524 components has been received. For example, it may check that all data from
525 neighboring elements has arrived to be able to continue integrating in time.
526 The signature of an `is_ready` method must be:
527

530 The `inboxes` is a collection of the tags passed to `receive_data` and are
531 specified in the iterable actions member type alias `inbox_tags`, which must be
532 a `tmpl::list`. The `inbox_tags` must have two member type aliases, a
533 `temporal_id` which is used to identify when the data was sent, and a `type`
534 which is the type of the data to be stored in the `inboxes`. The types are
535 typically a `std::unordered_map<temporal_id, DATA>`. In the discussed scenario
536 of waiting for neighboring elements to send their data the `DATA` type would be
537 a `std::unordered_map<TheElementIndex, DataSent>`. Having `DATA` be a
538 `std::unordered_multiset` is currently also supported. Here is an example of a
539 receive tag:

551 Once all of the `int`s have been received, the iterable action is executed, not
552 before.
553 
554 \warning
555 It is the responsibility of the iterable action to remove data from the inboxes
556 that will no longer be needed. The removal of unneeded data should be done in
557 the `apply` function.
558 
559 Iterable actions can change the type of the DataBox by adding or removing
560 elements/tags from the DataBox. The only requirement is that the last action in
561 the `action_list` returns a DataBox that is the same type as the
562 `initial_databox`. Iterable actions can also request that the algorithm no
563 longer be executed, and choose which action in the `ActionList`/`action_list` to
564 execute next. This is all done via the return value from the `apply` function.

722  * \defgroup TestingFrameworkGroup Testing Framework
723  * \brief Classes, functions, macros, and instructions for developing tests
724  *
725  * \details
726  *
727  * SpECTRE uses the testing framework
728  * [Catch](https://github.com/philsquared/Catch). Catch supports a variety of
729  * different styles of tests including BDD and fixture tests. The file
730  * `cmake/SpectreAddCatchTests.cmake` parses the source files and adds the found
731  * tests to ctest with the correct properties specified by tags and attributes.
732  *
733  * ### Usage
734  *
735  * To run the tests, type `ctest` in the build directory. You can specify
736  * a regex to match the test name using `ctest -R Unit.Blah`, or run all
737  * tests with a certain tag using `ctest -L tag`.
738  *
739  * ### Comparing double-precision results
740  *
741  * To compare two floating-point numbers that may differ by round-off, use the
742  * helper object `approx`. This is an instance of Catch's comparison class
743  * `Approx` in which the relative tolerance for comparisons is set to roughly
744  * \f$10^{-14}\f$ (i.e. `std::numeric_limits<double>::%epsilon()*100`).
745  * When possible, we recommend using `approx` for fuzzy comparisons as follows:
746  * \example
747  * \snippet TestFramework.cpp approx_default
748  *
749  * For checks that need more control over the precision (e.g. an algorithm in
750  * which round-off errors accumulate to a higher level), we recommend using
751  * the `approx` helper with a one-time tolerance adjustment. A comment
752  * should explain the reason for the adjustment:
753  * \example
754  * \snippet TestFramework.cpp approx_single_custom
755  *
756  * For tests in which the same precision adjustment is re-used many times, a new
757  * helper object can be created from Catch's `Approx` with a custom precision:
758  * \example
759  * \snippet TestFramework.cpp approx_new_custom
760  *
761  * Note: We provide the `approx` object because Catch's `Approx` defaults to a
762  * very loose tolerance (`std::numeric_limits<float>::%epsilon()*100`, or
763  * roughly \f$10^{-5}\f$ relative error), and so is poorly-suited to checking
764  * many numerical algorithms that rely on double-precision accuracy. By
765  * providing a tighter tolerance with `approx`, we avoid having to redefine the
766  * tolerance in every test.
767  *
768  * ### Attributes
769  *
770  * Attributes allow you to modify properties of the test. Attributes are
771  * specified as follows:
772  * \code
773  * // [[TimeOut, 10]]
774  * // [[OutputRegex, The error message expected from the test]]
775  * SPECTRE_TEST_CASE("Unit.Blah", "[Unit]") {
776  * \endcode
777  *
778  * Available attributes are:
779  *
780  * <table class="doxtable">
781  * <tr>
782  * <th>Attribute </th><th>Description </th>
783  * </tr>
784  * <tr>
785  * <td>TimeOut </td>
786  * <td>override the default timeout and set the timeout to N seconds. This
787  * should be set very sparingly since unit tests are designed to be
788  * short. If your test is too long you should consider testing smaller
789  * portions of the code if possible, or writing an integration test instead.
790  * </td>
791  * </tr>
792  * <tr>
793  * <td>OutputRegex </td>
794  * <td>
795  * When testing failure modes the exact error message must be tested, not
796  * just that the test failed. Since the string passed is a regular
797  * expression you must escape any regex tokens. For example, to match
798  * `some (word) and` you must specify the string `some \(word\) and`.
799  * </td>
800  * </tr>
801  * </table>
802  *
803  * \example
804  * \snippet Test_H5.cpp willfail_example_for_dev_doc

808  * You are able to test that a `static_assert` is being triggered using

812  * `Test_StaticAssertDataBox.cpp` must be added to the

823  * Specific compiler versions can be specified for which the regex changes.
824  * That is, the compiler version specified and all versions newer than that
825  * will use the regex, until a newer compiler version is specified. For

846 /*!
847  * \defgroup TimeGroup Time
848  * \brief Code related to the representation of time during simulations.
849  *
850  * The time covered by a simulation is divided up into a sequence of
851  * adjacent, non-overlapping (except at endpoints) intervals referred
852  * to as "slabs". The boundaries between slabs can be placed at
853  * arbitrary times. Slabs, as represented in the code as the Slab
854  * class, provide comparison operators comparing slabs agreeing with
855  * the definition as a sequence of intervals. Slabs that do not
856  * jointly belong to any such sequence should not be compared.
857  *
858  * The specific time is represented by the Time class, which encodes
859  * the slab containing the time and the fraction of the slab that has
860  * elapsed as an exact rational. Times are comparable according to
861  * their natural time ordering, except for times belonging to
862  * incomparable slabs.
863  *
864  * Differences in time within a slab are represented as exact
865  * fractions of that slab by the TimeDelta class. TimeDeltas are only
866  * meaningful within a single slab, with the exception that the ratio
867  * of objects with different slabs may be taken, resulting in an
868  * inexact floating-point result. Longer intervals of time are
869  * represented using floating-point values.
870  */
